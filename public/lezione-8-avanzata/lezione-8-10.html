Tipi di Tag

1. Lightweight Tag (Leggero)

# Crea tag lightweight (solo puntatore a commit):
git tag v1.0.0

# Leggero = semplicemente un nome per commit
# Non contiene metadata aggiuntiva
# Uso: marche temporanee, tag privati

# Caratteristiche:
‚úì Veloce da creare
‚úì Minimo overhead
‚úó No metadata (autore, data, messaggio)
‚úó Non firmabile

2. Annotated Tag (Annotato)

# Crea tag annotated (oggetto completo Git):
git tag -a v1.0.0 -m "Release version 1.0.0"

# Tag annotato = oggetto Git completo con:
# - Nome tagger
# - Email
# - Data
# - Messaggio
# - Opzionalmente firma GPG

# Caratteristiche:
‚úì Include metadata completa
‚úì Pu√≤ essere firmato (GPG)
‚úì Best practice per releases
‚úì Permanente e tracciabile

# CONSIGLIATO per release pubbliche

Creare Tags

# v1.0.0

# Mostra info tag:

Gestire Tags

# Lista tags:
git tag

# Lista con sorting:
git tag --sort=-version:refname  # Pi√π recenti prima

# Cerca tag con pattern:
git tag -l "v2.*"

# Mostra dettagli tag annotated:
git show v1.0.0

# Verifica tag firmato:
git tag -v v1.0.0

# Checkout tag (detached HEAD):
git checkout v1.0.0
# ‚ö†Ô∏è In detached HEAD state!

# Crea branch da tag:
git checkout -b hotfix-v1.0 v1.0.0

# Cancella tag locale:
git tag -d v1.0.0
git tag --delete v1.0.0

# Cancella tag remoto:
git push origin --delete v1.0.0
git push origin :refs/tags/v1.0.0  # Sintassi vecchia

# Rinomina tag (cancella e ricrea):
git tag new-name old-name
git tag -d old-name
git push origin :refs/tags/old-name
git push origin new-name

Push Tags a Remote

# Push singolo tag:
git push origin v1.0.0

# Push tutti i tags:
git push origin --tags

# Push solo annotated tags (Git 2.4+):
git push origin --follow-tags

# Configurazione auto-push tags:
git config --global push.followTags true
# Ora git push pusha anche tags annotati

# Fetch tags da remote:
git fetch --tags

# Pull con tags:
git pull --tags

# Esempio workflow release:
git tag -a v1.0.0 -m "Release 1.0.0"
git push origin v1.0.0
# Tag ora visibile su GitHub

GitHub Releases

GitHub Releases aggiunge UI e features aggiuntive ai Git tags.

Differenza Tag vs Release:

- Solo Git object
- Referenza a commit
- Visibile in git log

‚úÖ Best Practices - Tags e Releases

‚úÖ DO - Fare:

  • Usa Semantic Versioning (MAJOR.MINOR.PATCH)
  • Crea annotated tags per releases pubbliche
  • Scrivi release notes chiare e complete
  • Mantieni CHANGELOG.md aggiornato
  • Tagga solo commit su branch stable (main)
  • Test completi prima di taggare
  • Usa pre-release per alpha/beta
  • Firma tags per progetti critici (GPG)

‚ùå DON'T - Evitare:

  • Non cancellare tags pushati (altera storia)
  • Non usare tag come branch (sono immutabili)
  • Non taggare commit su feature branch
  • Non usare versioning inconsistente
  • Non saltare versioni (no v1.0 ‚Üí v1.3)
  • Non dimenticare di pushare tags
